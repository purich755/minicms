# Как пользоваться сервисом

Рабочая инструкция для того, кто продаёт и обслуживает. Инструкция для самого
клиента — отдельный файл, [КЛИЕНТУ.md](КЛИЕНТУ.md): его можно переслать как есть.

---

## Что вообще продаётся

Сайт заведения плюс личный кабинет, где владелец сам правит меню, акции и
новости. Без разработчика, без вёрстки, без «напишите мне, я поменяю цену».

Один движок обслуживает всех клиентов сразу. Новый клиент — это строка в базе,
а не новый проект: подключение занимает минуту, себестоимость почти нулевая.

Чужого клиент не увидит физически. Изоляция держится на политиках Postgres, а
не на фильтрах в коде, и проверяется прогоном `npm run test:rls` — 17 проверок
на настоящей базе.

## Подключить клиента

```bash
npm run tenant:create
```

Скрипт спросит три вещи и всё сделает сам:

| Вопрос | Что вводить | Пример |
|---|---|---|
| Название заведения | Как будет написано на сайте | `Кофейня «Флора»` |
| Адрес сайта | Латиницей, подсказку можно принять Enter'ом | `flora` |
| Почта владельца | На неё он будет входить | `owner@mail.ru` |

На выходе — готовый блок с адресом сайта, адресом панели, логином и паролем.
Его можно скопировать целиком и отправить клиенту вместе с [КЛИЕНТУ.md](КЛИЕНТУ.md).

**Пароль показывается один раз.** Не сохранил — придётся менять через Supabase →
Authentication → пользователь → Reset password.

Дальше клиент заходит на `/admin` и работает. Ничего донастраивать не нужно.

### Если адрес занят

Слаг — это адрес сайта, двух одинаковых быть не может. Скрипт скажет об этом
прямо. Занять `admin`, `api`, `www`, `app`, `static`, `public`, `assets`,
`_next` нельзя: они перекрыли бы служебные маршруты.

### Если клиент уже есть в системе

Такое бывает, когда у человека два заведения. Скрипт увидит знакомую почту,
переиспользует учётную запись и заведёт второй сайт. Пароль при этом **не
меняется** — отдавайте старый.

Оговорка: сейчас панель показывает владельцу только первое из его заведений.
Для второй точки нужен отдельный адрес почты, пока не сделан выбор активного
заведения.

## Показать демо

Заводить тенанта под каждую встречу не нужно — есть демо-сайт `flora-demo`.
Показывайте на нём: открыли сайт, зашли в панель, поменяли цену, обновили сайт —
цена другая. Это и есть весь смысл продукта, дальше объяснять обычно не
приходится.

Если встреча идёт вживую и заведение уже понятно — подключите его прямо при
клиенте за минуту. Это работает сильнее любой презентации.

## Резервные копии

**Supabase на бесплатном тарифе не делает копий вообще.** Клиент, который
потратил вечер на меню, потеряет его без следа. Поэтому копии — на нас.

```bash
npm run backup
```

Кладёт в `backups/2026-07-21_1530/`:

- `data.json` — все строки: заведения, меню, акции, новости, настройки
- `media/` — картинки из Storage
- `README.txt` — что внутри и как вернуть

Папка `backups/` в git не попадает: там живые данные клиентов.

### Вернуть из копии

```bash
npm run restore -- backups/2026-07-21_1530          # показать, что приедет
npm run restore -- backups/2026-07-21_1530 --yes    # выполнить
```

Без `--yes` ничего не пишется — только печатается, сколько строк и картинок
приедет. Строки идут через upsert по id: существующие обновляются, новые
создаются. Запускать дважды подряд безопасно.

**Что restore не делает:** не удаляет то, чего нет в копии. Добавили пять
позиций после снимка — они останутся. Так задумано: молча стереть данные хуже,
чем оставить лишнее.

**Чего в копии нет:** паролей. Их нельзя выгрузить, они хранятся хешами. Для
обычной беды это неважно — учётные записи на месте, вернуть надо содержимое.
Если потерян весь проект Supabase, владельцев заводят заново через
`npm run tenant:create`, а потом восстанавливают содержимое.

### Чтобы копии снимались сами (Windows)

Планировщик заданий → Создать простую задачу:

- Триггер: ежедневно, время нерабочее
- Действие: запуск программы
- Программа: `cmd.exe`
- Аргументы: `/c cd /d "G:\claude project\mini-cms" && npm run backup`

Раз в месяц копируйте папку `backups/` куда-то ещё — на флешку или в облако.
Копия, которая лежит на том же диске, что и всё остальное, спасает от ошибки
клиента, но не от смерти диска.

## Черновики и предпросмотр

Клиент может готовить контент, не показывая его гостям:

- **новость** — снять галочку «Опубликовать», она станет черновиком;
- **акция** — снять «Активна», либо поставить дату начала в будущем;
- **позиция меню** — снять «В наличии».

Чтобы посмотреть, как это будет выглядеть, в панели есть кнопка
**Предпросмотр**. Она открывает настоящий сайт, но вместе с неопубликованным.
Скрытое отмечено жёлтой меткой с причиной: «Черновик», «Нет в наличии», «Ещё
не началась», «Закончилась», «Выключена». Внизу висит плашка с кнопкой
«Выйти».

Предпросмотр видит только владелец и только свой сайт. Гость, случайно
получивший ссылку, увидит обычную версию — проверка идёт по тому, чей это
сайт, а не по одному факту включённого режима.

## Обслуживание

| Что случилось | Что делать |
|---|---|
| Клиент забыл пароль | Supabase → Authentication → пользователь → Reset password. Своей почты для писем пока нет, поэтому руками. |
| Клиент просит свой домен | Добавить домен в Vercel, прописать его в `tenants.custom_domain`. Подробности в [DEPLOY.md](DEPLOY.md). |
| «Сайт не открывается» | `npm run check:supabase` — ключи, схема, гранты. |
| «Поменял, а на сайте старое» | Сохранение сбрасывает кеш сразу. Если старое осталось — проверьте, что правили нужное заведение. |
| Правили политики или схему | `npm run test:rls` обязательно. Регрессия в изоляции тихая: всё работает, просто один клиент видит другого. |

## Перед первым платящим клиентом

- [ ] `SUPABASE_SERVICE_ROLE_KEY` заполнен в `.env.local` (нужен скриптам)
- [ ] Прежний секретный ключ, который утёк в переписку, **отозван** в Supabase
- [ ] Репозиторий на GitHub переведён в private
- [ ] `npm run backup` отработал хотя бы раз, файл на месте
- [ ] Автоматическая задача на копии заведена
- [ ] `npm test` зелёный
- [ ] Демо-показ отрепетирован на `flora-demo`

## Что стоит доделать дальше

По убыванию пользы:

1. **Своя почта для писем (SMTP).** Пока клиент не может восстановить пароль
   сам — чинить приходится руками. Это первое, что начнёт раздражать.
2. **Выбор активного заведения.** Сейчас владелец с двумя точками видит только
   первую.
3. **Оплата подписки.** Пока учёт вручную.
4. **Картинка в шапку сайта.** Требует миграции: поля под неё ещё нет.
