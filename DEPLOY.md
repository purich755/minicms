# Как выложить сайт в интернет

Пошагово, от локального проекта до работающего адреса, который можно
отправить клиенту. Дальше правки кода будут выкатываться сами при `git push`.

Всё, что здесь есть, проверено на этом проекте, кроме шагов внутри аккаунтов
GitHub и Vercel — их делаешь ты.

---

## 1. Репозиторий на GitHub

Проект уже под git, но лежит только на твоём диске. Нужен удалённый
репозиторий — из него Vercel будет забирать код.

1. [github.com/new](https://github.com/new) → имя, например `mini-cms`.
2. **Visibility: Private.** Публичным делать незачем: в истории нет секретов,
   но и выставлять движок наружу смысла нет.
3. **Не** ставь галки на README, .gitignore и лицензию — они уже есть, иначе
   получишь конфликт при первом пуше.
4. Скопируй адрес репозитория и в терминале, из папки `mini-cms`:

```powershell
git remote add origin https://github.com/ТВОЙ_ЛОГИН/mini-cms.git
git push -u origin master
```

Проверь после пуша: `.env.local` в репозитории быть не должно. Он в
`.gitignore`, но убедиться стоит — в нём ключи.

## 2. Проект на Vercel

1. [vercel.com/new](https://vercel.com/new) → Import Git Repository → выбрать
   репозиторий.
2. Framework Preset определится как Next.js сам.
3. **Root Directory оставить пустым.** Репозиторий и есть корень проекта.
4. Не нажимай Deploy сразу — сначала переменные, иначе первая сборка упадёт.

### Переменные окружения

Environment Variables, три штуки. Значения — те же, что в твоём `.env.local`:

| Имя | Значение | Примечание |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://…supabase.co` | **без** `/rest/v1` на конце |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `sb_publishable_…` | только publishable |
| `NEXT_PUBLIC_ROOT_DOMAIN` | пусто | заполняется на шаге 4 |

Отметь все три для Production, Preview и Development.

> Секретный ключ (`sb_secret_…`) сюда не добавляй. Переменные с префиксом
> `NEXT_PUBLIC_` попадают в javascript, который скачивает каждый посетитель.

Теперь Deploy. Через пару минут получишь адрес вида
`mini-cms-xxxx.vercel.app`.

## 3. Проверка после первого деплоя

Открой по порядку:

- `/` — витрина сервиса
- `/flora-demo` — демо-кофейня с меню, акциями и новостями
- `/flora-demo/menu` — 9 позиций, «Тирамису» скрыт (снят с продажи)
- `/flora-demo/news` — 2 новости, черновик скрыт
- `/admin` — форма входа, дальше твоя почта и пароль

Если публичные страницы пустые или отдают 404 — почти наверняка переменные
окружения. Проверь URL: лишний `/rest/v1` на конце ломает всё молча.

## 4. Свой домен

Пока адрес — `*.vercel.app`. Клиенту такое не покажешь.

### Домен сервиса

1. Vercel → проект → Settings → Domains → добавить `твойдомен.рф`.
2. У регистратора прописать записи, которые покажет Vercel.
3. Там же добавить **`*.твойдомен.рф`** — это включит поддомены для клиентов.
   В DNS понадобится запись `*` типа CNAME на Vercel.
4. В переменных окружения выставить `NEXT_PUBLIC_ROOT_DOMAIN=твойдомен.рф`
   и передеплоить.

После этого сайт клиента открывается тремя способами сразу:

| Адрес | Когда используется |
|---|---|
| `твойдомен.рф/flora-demo` | работает всегда, ничего настраивать не надо |
| `flora-demo.твойдомен.рф` | после wildcard-домена |
| `flora-cafe.ru` | свой домен клиента, см. ниже |

### Домен клиента

1. Vercel → Domains → добавить `flora-cafe.ru`.
2. Клиент у своего регистратора прописывает записи от Vercel.
3. В базе связать домен с тенантом:

```sql
update public.tenants
set custom_domain = 'flora-cafe.ru'
where slug = 'flora-demo';
```

Резолв по своему домену идёт запросом в базу, но результат кешируется, так
что на каждый просмотр страницы запроса нет.

## 5. Как теперь вносить изменения

**Контент клиента** — через `/admin`. Правка видна на сайте сразу: админка
сбрасывает кеш нужного раздела.

**Код** — обычный цикл:

```powershell
npm run dev          # правишь, смотришь на localhost:3200
npm test             # логика + изоляция тенантов
git add -A
git commit -m "что изменил"
git push
```

Vercel соберёт и выкатит сам за пару минут. Ветки кроме `master`
разворачиваются на отдельные превью-адреса — на них можно посмотреть правку,
не трогая боевой сайт.

**Схема базы** — миграции применяются вручную через SQL Editor. Файл кладётся
в `supabase/migrations/` с датой в имени, чтобы порядок был очевиден, и после
применения обязательно:

```powershell
npm run check:supabase
```

Он сверит колонки с `lib/types.ts` и скажет, если схема разошлась с кодом.

---

## Что стоит закрыть до первого платящего клиента

- **Бэкапы.** На бесплатном тарифе Supabase их нет вообще. До заливки
  реального контента настрой регулярный дамп.
- **Восстановление пароля.** Сейчас забытый пароль чинится только руками
  через дашборд Supabase. Нужен свой SMTP.
- **Сон проекта.** Бесплатный Supabase засыпает после 7 дней без запросов.
  Для сайта с посетителями не проблема, для «тихого» клиента — да.

## Грабли, на которые уже наступали

- **`/rest/v1` в конце URL Supabase.** Всё отвечает 404 и выглядит как
  «миграции не применены», хотя схема на месте.
- **Секретный ключ вместо publishable.** Он обходит RLS: попав в
  `NEXT_PUBLIC_`, уезжает в браузер каждого посетителя. Такой ключ надо
  считать сгоревшим и отзывать.
- **Забытая галка «Auto Confirm User».** Пользователь создан, а войти нельзя,
  и внятной ошибки нет.
- **`INSERT` в SQL Editor** всегда пишет «Success. No rows returned»,
  независимо от того, вставилось ли что-то. Проверять отдельным `select`.
