# mini-cms — мультитенантная CMS для сайтов локального бизнеса

Одна кодовая база и одна БД обслуживают много клиентов сразу. Клиент (кафе,
барбершоп, салон) — это **тенант**. У тенанта есть публичный сайт с его меню,
акциями и новостями, и админка, где владелец сам всё это правит. Новый клиент —
это новая строка в БД, а не новый проект.

## Стек

- Next.js 16 (App Router, TypeScript), Turbopack по умолчанию
- Supabase: Postgres + Auth + Storage
- Tailwind CSS 4 (конфиг в CSS, файла `tailwind.config.js` нет)
- Деплой — Vercel
- Интерфейс и контент — русский, цены в рублях (₽)

## Незыблемые правила

1. **Публичные страницы рендерятся на сервере.** Браузер посетителя никогда не
   ходит в Supabase напрямую — только сервер Vercel. Это нужно для скорости и
   доступности из России. В `app/(public)/**` не должно появиться `'use client'`
   с запросами к БД.
2. **Изоляция тенантов держится на трёх уровнях**, и все три обязательны:
   `tenant_id` в каждой контентной таблице; RLS-политики в Postgres; резолв
   тенанта по адресу. Фильтрация по `tenant_id` в коде — это удобство, а не
   защита. Защита — RLS.
3. **Service role key живёт только на сервере.** Он обходит RLS целиком. Никогда
   не импортируется в файл с `'use client'` и не читается через `NEXT_PUBLIC_*`.
   Файлы, которым нельзя в браузер, начинаются с `import 'server-only'`.
4. **Публичное чтение — только опубликованное**: `is_published = true`,
   `is_active = true`. Это тоже уровень RLS, а не только `.eq()` в запросе.
5. **Вся работа с БД — через `lib/supabase`.** В компонентах не создаём клиентов
   и не пишем запросы напрямую.
6. Секреты — только через `.env`, в коде не хардкодятся.

## Три клиента Supabase, не путать

| Файл | Роль в БД | Где применяется |
|---|---|---|
| `lib/supabase/public.ts` | `anon`, сессии нет вообще | публичные страницы `app/(public)/**` |
| `lib/supabase/server.ts` | `authenticated` по cookies | админка: серверные компоненты, server actions |
| `lib/supabase/client.ts` | `authenticated` в браузере | клиентские компоненты админки |

Публичные страницы ходят в базу **без сессии** — это не оптимизация, а часть
модели безопасности. RLS-политики публичного чтения выписаны на роль `anon`,
поэтому владелец одного кафе не прочитает строки другого даже из-под своей
сессии. Побочный плюс: ответ не зависит от зрителя, страницу можно кешировать
и сбрасывать по тегу.

Обращения к таблице `tenants` обязаны перечислять колонки явно
(`.select('id, slug, name, custom_domain')`). Колонки `plan` и `owner_user_id`
наружу не выданы грантами, и `select('*')` упрётся в permission denied.

## Структура

```
app/(public)/[tenant]/   публичные сайты клиентов (SSR)
app/admin/(auth)/login/  страница входа — без защиты
app/admin/(protected)/   всё остальное в админке — layout редиректит гостей
app/admin/actions.ts     server actions входа и выхода
components/{admin,public,ui}
lib/supabase/            client.ts · server.ts · public.ts · proxy.ts · env.ts
lib/auth.ts              текущий пользователь и его тенант, requireTenant()
lib/tenant.ts            резолв: домен/слаг → тенант (Фаза 4)
lib/types.ts             типы схемы БД
supabase/migrations/     SQL: таблицы, RLS, политики Storage
proxy.ts                 защита /admin и продление сессии (бывший middleware.ts)
```

Группы маршрутов `(auth)` и `(protected)` нужны, чтобы защищающий layout не
оборачивал саму страницу входа: иначе редирект на /admin/login зациклился бы.

## Правило для server actions

Каждое действие, которое читает или пишет данные тенанта, начинается с

```ts
const tenant = await requireTenant()
```

Не потому, что так аккуратнее, а потому что **proxy до server actions не
дотягивается**: они приходят POST-запросом на маршрут, где объявлены, и
матчер proxy их может не накрыть. Проверка внутри действия — единственная,
на которую можно положиться. Плюс RLS в базе как последний рубеж.

## Проверка изоляции тенантов

```
npm run test:rls          17 проверок: что владелец видит только своё
npm run test:rls:mutate   + доказательство, что проверки умеют краснеть
```

Прогон поднимает настоящий Postgres (PGlite, WASM), накатывает все миграции из
`supabase/migrations/` и выполняет `supabase/checks/02_rls_check.sql` — тот же
файл, что запускается руками в Supabase SQL Editor. Ни интернета, ни Supabase
не требуется.

**Запускать после любой правки политик или схемы.** Регрессия в RLS тихая:
приложение продолжает работать, просто один клиент начинает видеть данные
другого. Если добавляешь таблицу с `tenant_id` — добавь и проверку на неё.

Заглушки `auth`/`storage` для локального прогона лежат в
`supabase/checks/_supabase_stubs.sql`; в Supabase этот файл не применяется.

## Особенности Next.js 16 (отличия от того, что помнит модель)

- `middleware.ts` переименован в **`proxy.ts`**, экспорт функции — `proxy`.
  Runtime у него `nodejs`, edge не поддерживается.
- `params` и `searchParams` в страницах и лейаутах, а также `cookies()` и
  `headers()` — **только асинхронные**. Синхронный доступ удалён.
- `revalidateTag(tag)` требует второй аргумент: `revalidateTag('menu', 'max')`.
- Для «сохранил в админке — сразу видно на сайте» правильный инструмент —
  **`updateTag(tag)`** из `next/cache`: он работает только внутри server actions
  и даёт read-your-own-writes без отдачи устаревшего контента.
- `next lint` удалён, линт запускается как `npm run lint` (ESLint CLI).
- Перед тем как писать код по незнакомому API, читай
  `node_modules/next/dist/docs/` — там документация ровно этой версии.

## Порядок работы

Проект строится по фазам, каждая проверяется отдельно; после фазы — отчёт и
пауза, дальше только с подтверждения. Фаза 0 — каркас. Фаза 1 — БД и RLS.
Фаза 2 — вход и каркас админки. Фаза 3 — CRUD. Фаза 4 — публичный сайт.
Фаза 5 — мультидомены. Фаза 6 — полировка и деплой.

Архитектурные развилки (схема таблицы, способ авторизации, схема доменов) не
решаются молча — выносятся на обсуждение.
